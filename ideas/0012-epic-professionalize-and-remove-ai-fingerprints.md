# Professionalize the plugin — remove AI fingerprints, raise codebase standards

Raw idea:

As part of the professionalizing of the app, go through the complete app and remove all signs of ai generation by Claude or any other tool.

Refined direction (from shaping chat):

Two parts, done together but scoped separately:

- **Part A — Remove AI fingerprints.** A pass across the repo to strip tool-identifying metadata (references to Claude Code, AI-specific file markers, generator tags) and clean obvious AI-style tells from user-visible copy. Cosmetic pass, high visibility, low risk.
- **Part B — Raise codebase standards.** The deeper "professionalize" work: consistent style, linted formatting, real test coverage, proper README, changelog hygiene, contribution guidelines, privacy policy, accessibility audit, security audit. The kind of thing a prospective buyer or serious customer would check before trusting the product.

The two parts support each other: good documentation and consistent style make the codebase look like it's been cared for, which is what "professionalizing" actually means. "No AI fingerprints" falls out naturally as a sub-item.

## Why this is an epic

Part A alone is a single-sprint task. Part B is a multi-sprint quality uplift that touches the whole repo. Together they're an epic because the Part B work is genuinely substantial (tests, lint, docs, audits) and because the two should be planned together — a single readthrough pass can hit both.

## Part A — Remove AI fingerprints

### What's in the repo right now (concrete findings from a grep pass)

- **`ideas/README.md`, `specs/README.md`, `DEVOPS.md`** mention "Claude Code" by name, repeatedly. 20+ direct references in these files alone. These are internal/dev-facing docs but they're in the public repo, so anyone browsing GitHub sees them.
- **`ideas/0002-feat-access-control-roles-overview.md`** contains the heading "Answer (not an idea — Claude Code should record this and move on)" — that's an instruction to a tool embedded in a product file.
- **`languages/talenttrack.pot`** has `X-Generator: TalentTrack i18n` in its header. Not AI-identifying but worth standardizing (most `.pot` tools use canonical generator strings like `WP-CLI 2.x`).
- **Em dash density.** 928 em dashes across the non-vendor files (76 in `readme.txt` alone, 39 in `CHANGES.md`). Em dashes aren't inherently AI — plenty of human writers use them — but at this density they read as a tell. A controlled reduction to ~30–40% of current count, replacing most with commas, colons, or sentence breaks, would remove the signal without making the prose stiff.
- **Changelog + readme copy style.** Heavy use of "— NEW:" / "— CHANGED:" bullets, parenthetical explanations, and the "short declarative clause, em dash, expansion" rhythm across `readme.txt` and `CHANGES.md`. Characteristic LLM pacing. Readable, but uniform in a way that human-written changelogs usually aren't.
- **No obvious `@generated` tags, "Generated by Claude" comments, or tool-signed commits in the code itself** — that's a clean win, nothing to strip on the PHP side.

### Scope of changes

1. **Strip tool-specific references from public docs.**
   - `ideas/README.md`, `specs/README.md`, `DEVOPS.md`: rewrite to describe the *process* (shaping ideas, promoting specs, cutting releases) without naming the tool used to run it. "Claude Code reads `ideas/NNNN-*.md`…" becomes "Run the shaping workflow against `ideas/NNNN-*.md`…" Same information, tool-agnostic phrasing. This keeps the workflow intact — you can still use Claude Code, you just don't advertise it.
   - `ideas/0002-*`: remove the embedded tool instruction from the heading.
   - `DEVOPS.md`: the section "When to ask Claude for help in chat vs in Claude Code" gets rewritten or moved to a private `CONTRIBUTING.internal.md` that isn't in the public repo.
2. **Normalize generator metadata.**
   - `.pot` header: regenerate via `wp-cli i18n make-pot` (this gets done anyway in idea #0010 — the multi-language one) so the `X-Generator` becomes the canonical WP-CLI string.
   - Any future templates that set custom generator strings should just omit the line.
3. **Copy editing pass on user-visible prose.**
   - Files: `readme.txt`, `CHANGES.md`, `DEVOPS.md`, everything in `docs/`, everything in `docs/nl_NL/`, and the translated docs added in idea #0010.
   - Reduce em dashes to ~30–40% of current density. Vary sentence rhythm. Break up uniform "NEW: / CHANGED: / FIXED:" bullet patterns with occasional prose paragraphs where context helps.
   - Watch for "delve into", "leverage", "seamlessly", "comprehensive", "robust", "unlock the power of", "it's worth noting that", "furthermore", "moreover" — low count in current docs, easy to catch.
   - This is copyediting, not rewriting. Don't change meaning, don't add/remove features, don't alter changelog content.
4. **Leave the code alone.** No rewriting of PHP, no reshuffling of comments, no removing defensive error handling. Part A is docs + metadata only. Code quality improvements are Part B's domain and they go through linting, not vibes.

### What Part A explicitly does NOT do

- Does not ban continued use of AI tools internally. The goal is that the *public-facing surface* doesn't advertise the tooling, not that the team changes how they work.
- Does not try to "de-AI-ify" PHP code based on stylistic instinct. That's a trap (you end up stripping comments and error handling to make code "look less AI").
- Does not remove `CLAUDE.md` or similar config files if they exist at the repo root *and are gitignored* — they're developer local config, invisible to anyone browsing the repo.

## Part B — Raise codebase standards

The substantive part. Four workstreams, each roughly a sprint.

### B1 — Code style and linting

- **PHP_CodeSniffer + WordPress Coding Standards ruleset** (`WordPress-Extra`), configured via `phpcs.xml.dist` at the repo root. Run on every PR.
- **PHP-CS-Fixer** for auto-fixable issues.
- **ESLint + Prettier** for `assets/js/` and any future JS.
- **Pre-commit hook** (via a lightweight tool like `pre-commit` or just a shell script) so style violations never reach a commit.
- One-off baseline: auto-fix everything fixable, manually resolve the rest, land as a single "style baseline" commit so the git history stays readable.

### B2 — Test coverage

- Current state: `composer.json` exists but (per a quick look) there's no `phpunit.xml` and no `tests/` directory. That's the biggest gap.
- Set up PHPUnit with `wp-phpunit` or Brain Monkey depending on how much WP-context the tests need.
- Start with integration tests for the REST endpoints under `includes/REST/` — highest-leverage coverage for a plugin.
- Target: ~40% line coverage for modules handling writes (Players, Teams, Evaluations, Goals, Sessions). Higher targets later; 40% is the "meaningful coverage, not aspirational" bar.
- GitHub Actions workflow to run tests on every PR. No test run, no merge.

### B3 — Documentation

- **README.md at repo root.** Currently the README is buried in `plugin-update-checker/` (that's the vendored lib's readme). Need a real top-level README: what the plugin does, who it's for, how to install, link to docs, link to specs/, changelog link, license, contribution pointer.
- **CONTRIBUTING.md.** How to propose ideas (link to `ideas/`), how to shape specs, PR expectations, coding standards, how to run tests. Replaces the tool-specific phrasing that Part A strips out of `DEVOPS.md`.
- **SECURITY.md.** How to report a vulnerability. GitHub renders this as a security policy banner — expected on any serious public repo.
- **CODE_OF_CONDUCT.md.** Standard Contributor Covenant. Another expected artifact.
- **CHANGELOG standardization.** `CHANGES.md` and `readme.txt` have drifted — they're mostly consistent but duplicated. Consider a `CHANGELOG.md` that follows Keep a Changelog, and have `readme.txt`'s `== Changelog ==` section generated from it during build.
- **Inline PHPDoc audit.** Public methods on classes in `src/Modules/` should all have `@param`, `@return`, `@throws`. PHPCS can enforce this.

### B4 — Security + privacy + accessibility

- **Security audit.** Pass over all `$_POST`, `$_GET`, `$_REQUEST` usage — verify nonces, capability checks, `wp_unslash`, sanitization, prepared statements. A one-time pass, then PHPCS rules to prevent regression.
- **Privacy policy.** If the plugin collects any player data (it does — names, evaluations, goals), there's a GDPR story to tell. Draft a privacy statement users can include on their site, document what data is stored, how long, with what controls for deletion.
- **Accessibility audit.** Admin pages against WCAG 2.1 AA. Tab order, ARIA labels on the drag-reorder lookup tables (SortableJS is notoriously bad for a11y by default), contrast on custom-branded UI. This is increasingly a purchasing criterion for public-sector and EU buyers.
- **License file cleanup.** `license.txt` or `LICENSE` at root, GPL-2.0+ matching the plugin header. Quick to verify.

## Decomposition / rough sprint plan

1. **Sprint 1 — Part A (fingerprints).** The cosmetic pass. Doc rewrites, em dash reduction, metadata normalization. Low risk, high visibility, ship fast.
2. **Sprint 2 — B1 (lint + style).** PHPCS, CS-Fixer, ESLint, pre-commit hook, baseline fix. One big PR for the baseline.
3. **Sprint 3 — B2 (tests).** PHPUnit setup, REST endpoint tests for write paths, CI integration.
4. **Sprint 4 — B3 (docs).** README, CONTRIBUTING, SECURITY, CODE_OF_CONDUCT, CHANGELOG consolidation, PHPDoc pass.
5. **Sprint 5 — B4 (security/privacy/a11y).** Audit pass, fixes, privacy statement draft, accessibility fixes.

Part A should go first and alone — it's fast, and it's the thing most visible to anyone looking at the repo. Parts B1–B4 can reorder based on what you think matters most.

## Open questions

- **Does "tool-agnostic phrasing" go far enough, or do you want `DEVOPS.md` to stop describing a shaping workflow entirely?** My read: describe the workflow, don't name the tool. A workflow with specs + ideas folders is a *good* thing to document — it's a signal of a thought-through process. The only change is it shouldn't read as "here's how I use Claude."
- **How strict on em dashes?** Full removal reads stiff. Half removal is probably right. I'd pick a sampling of 20 em dashes at random from `readme.txt`, rewrite each one, and see how it reads — if it reads natural, scale that up.
- **Public vs private contributor docs.** If you want to describe the internal workflow in detail (tool use, shaping prompts, etc.), put it in a file that isn't in the public repo — `CONTRIBUTING.internal.md` in a private fork, or just a notes file in your local machine. The public `CONTRIBUTING.md` describes the process at the level a new external contributor would need.
- **Test coverage target.** 40% is my proposed floor. Some shops want 80%+. Depends on whether tests are "I want to catch regressions" or "I want buyers to see a coverage badge". Badge-driven testing is low value; regression-driven testing at 40–60% is real value.
- **Do the translated docs (`docs/nl_NL/` + the ones added in #0010) get the same copy editing pass?** They should. The em-dash + "uniform bullet pacing" tells translate surprisingly well into other languages and are equally identifiable there.
