# Translator brief

If you're translating TalentTrack into a new locale (or filling in the empty FR/DE/ES skeletons that #0010 ships), read this first.

## What you're translating

`languages/talenttrack-<locale>.po` ‚Äî one file per locale. Each entry is a `msgid` (the canonical English source string) followed by a `msgstr` (your translation). Empty `msgstr ""` means "no translation; render the English instead" ‚Äî that's the WordPress runtime fallback. Leaving entries empty is a valid intermediate state.

Compiled `.mo` files are auto-generated by CI on merge to main (`.github/workflows/translations.yml`). You don't run `msgfmt` locally.

## Tone ‚Äî informal vs. formal per surface

TalentTrack uses **mixed formality** by surface, not one tone across the product. The choice depends on who's reading the string and in which view.

| Surface | French | German | Spanish |
|---|---|---|---|
| Player dashboard, evaluations view, goals, my profile | Tu | Du | t√∫ |
| Coach-facing session entry, attendance, player browsing | Tu | Du | t√∫ |
| HoD-facing reports, decision panels | Vous | Sie | t√∫ |
| Admin / Settings, config pages, role management | Vous | Sie | t√∫ |
| System-level errors, migration messages | Vous | Sie | t√∫ |
| Emails / letter templates to parents | Vous | Sie | usted |

**Spanish (`es_ES`)**: the formal/informal distinction is less rigidly tied to context than in French/German. Default `t√∫` throughout for warmth and approachability. Reserve `usted` for the most formal external-facing communications ‚Äî denial letters to parents, GDPR notices, anything legally formal.

**Reasoning**: the player and coach surfaces are conversational; the operator/admin surfaces deal with consequence (deleting data, granting permissions, sending letters). The tone should match.

## How to know which surface a string belongs to

Three ways:

1. **Translator comments** ‚Äî `/* translators: admin-facing */` or `/* translators: player surface */` lines preceding `__()` calls in PHP. These appear in the `.po` as `#. translators: ‚Ä¶` lines above the msgid. Read them.

2. **Source file path** in the `#: ‚Ä¶` reference comment ‚Äî `src/Modules/Authorization/Admin/RolesPage.php` is admin-facing; `src/Shared/Frontend/PlayerDashboardView.php` is player-facing. The path tells the surface.

3. **Ask if unsure** ‚Äî if a msgid could fit either surface and the comment is missing, lean formal (`Vous` / `Sie` / `usted`). Mistakenly-formal strings on a player surface read slightly stiff; mistakenly-informal strings on an admin surface read unprofessional. Stiff > unprofessional.

If you find a string whose tone genuinely isn't decidable from the comment + path, file a PR adding a `/* translators: ‚Ä¶ */` line so future translators don't have to make the same call.

## Names and proper nouns

- **Don't translate** entity names that are part of the product vocabulary: `TalentTrack`, `Spond`, `Freemius`, `WhatsApp`, `WordPress`, `MySQL`. These ship as-is in every locale.
- **Don't translate** technical terms operators interact with that the codebase uses literally: `URL`, `REST API`, `JSON`, `XLSX`, `CSV`, `nonce`, `wp-config.php`. Translating "REST API" to "API REST" looks fine in some locales (FR/ES) and odd in others (DE). Default: leave as-is.
- **Do translate** football/sports vocabulary that has a natural local equivalent: "Goalkeeper" ‚Üí "Gardien de but" (FR) / "Torwart" (DE) / "Portero" (ES). When in doubt, mirror what local football associations and broadcasters use, not what a generic translator would output.

## Plurals

`.po` supports plural forms via `msgid_plural` + `msgstr[0]` / `msgstr[1]` / etc. Each locale declares its plural rule in the file header (`Plural-Forms:` line). The skeletons ship with:

- French: `nplurals=2; plural=(n > 1);` (singular for 0 + 1, plural for 2+)
- German: `nplurals=2; plural=(n != 1);` (singular for 1, plural for everything else ‚Äî same as English)
- Spanish: `nplurals=2; plural=(n != 1);` (same as German)

Don't change those rules unless you know what you're doing. The English source uses `_n()` for plural-aware strings ‚Äî translate each `msgstr[N]` block correspondingly.

## Strings with placeholders

`%s`, `%d`, `%1$s`, etc. ‚Äî keep them in the same order as the English when grammar permits, reorder via positional tokens (`%1$s`, `%2$s`) when grammar requires. Example:

```po
msgid "Welcome, %1$s. You have %2$d unread messages."
msgstr "Bienvenue, %1$s. Vous avez %2$d messages non lus."
```

If a placeholder needs to land in a different position than the English, ALWAYS use the positional `%1$s` form so the `printf` call doesn't break. Never drop a placeholder; never add one that wasn't in the source.

## HTML in strings

Some msgids include inline HTML ‚Äî `<a href="‚Ä¶">Click here</a>`, `<strong>‚Ä¶</strong>`, etc. Translate the visible text; leave the tags intact and in the same positions. The HTML is part of the contract the rendering code expects.

## What does NOT belong in `.po` (since #0090 Phase 6)

Data-row strings ‚Äî lookup labels (`Goalkeeper`, `Pending`, etc.), evaluation-category names, role labels, functional-role labels ‚Äî live in the `tt_translations` database table, not in `.po`. See `docs/i18n-architecture.md` for the split.

If you find a `msgid` in the `.po` that's clearly a database-row label (e.g. `"Goalkeeper"` for the position lookup), translate it anyway ‚Äî those msgids stay as a fallback path until the operator authors the per-row translation via the admin Translations form. Phase 8 of #0090 chose to leave the migrated msgids in `.po` as belt + braces; pruning them is a possible cleanup once telemetry confirms zero callers hit the gettext fallback in practice.

## Workflow

1. Pull the latest `main` and check out a `chore/i18n-<locale>` branch.
2. Open `languages/talenttrack-<locale>.po` in Loco / Poedit / your text editor.
3. Translate `msgstr` entries. Save.
4. Push the branch and open a PR. The `Validate .po syntax` CI job runs `msgfmt --check`; merge when green.
5. On merge to main, the `Auto-compile .po ‚Üí .mo` workflow regenerates the `.mo` automatically and commits it.

If you're working through a large batch over multiple sessions, partial commits are fine ‚Äî empty msgstrs fall back to English at runtime, so the product never displays a broken page.

## Estimated effort

For the `~1000 msgid` UI translation (per #0010 spec):

- Machine-translate as a first pass via DeepL / Google Translate ‚Äî fast, ~1 hour for the whole `.po`.
- Native-speaker review pass ‚Äî **~15-25 hours per language**. This is the expensive step: reading every string in context, fixing obvious machine-translation mistakes (especially on tone), checking the placeholder ordering.
- Docs translation (19 docs in `docs/`, ~500 words each, into 3 locales √ó 19 files = 57 files) ‚Äî separate, **~30-60 hours total**.

Plan accordingly. The `.po` is the smaller deliverable and ships first; docs follow as calendar-time permits.
